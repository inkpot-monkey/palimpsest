* 🌌 NixOS Configuration

A structured, reproducible NixOS configuration managed with *Flakes*, *Home Manager*, and *SOPS* for secret management.

** 📂 Repository Structure

The configuration is organized to be modular and explicit, utilizing *Profiles* to encapsulate features.

#+begin_src
├── flake.nix             # Entry point. Defines global inputs and outputs.
├── lib/                  # Flake-wide library helpers (mkSystem, mkPiSystem, mkPkgs)
├── hosts/                # System-level configurations (NixOS)
│   ├── default.nix       # Entry for nixosConfigurations
│   └── [hostname]/       # machine-specific (e.g., stargazer, kelpy)
├── users/                # User definitions & Home Manager
│   ├── default.nix       # Entry for homeConfigurations & self.users modules
│   └── [username]/       # user-specific (e.g., inkpotmonkey, general)
├── modules/              # Reusable modules & Profiles
│   ├── nixos/
│   │   ├── services/     # Public/Reusable service modules (git-annex, jmap)
│   │   └── profiles/     # Feature bundles (desktop, virtualization, gaming)
│   ├── home-manager/     # Home Manager modules
│   └── shared/           # Shared logic (overlays)
├── pkgs/                 # Custom packages
├── parts/                # Flake parts (dev shells, checks)
└── secrets/              # Encrypted secrets (sops-nix)
#+end_src

** 🧠 Core Concepts

*** 1. Feature Profiles
We avoid monolithic configurations by using *Profiles* in ~modules/nixos/profiles/~.
- *base*: Core Nix settings, GC, and experimental features.
- *common*: Networking, locale, timezone, and base packages.
- *desktop*: Graphical environment and desktop-wide tools.
- *virtualization*: Docker, Libvirtd, and bridge networking.
- *ai*: ROCm-enabled Ollama and AI tools.
- *gaming*: Steam, Gamemode, and hardware tweaks.

Hosts simply import what they need from ~self.nixosProfiles.~.

*** 2. Clean User Imports
Users are exposed as first-class NixOS modules through ~self.users.[username]~.
- This manages the system-level account (groups, shell, authorized keys).
- It automatically triggers the integrated **Home Manager** configuration.
- Keeps all user-specific code encapsulated within the ~users/~ directory.

*** 3. Standardized Lib Helpers
We use a centralized library in ~lib/default.nix~ to reduce host boilerplate:
- *mkSystem*: Standard NixOS system (x86_64).
- *mkPiSystem*: Specialized helper for Raspberry Pi images, utilizing ~nixos-raspberrypi~.

*** 4. GRANULAR Keys
User SSH keys are decentralized. Instead of a global key list, each user manages their own keys in their respective directory (~users/[username]/keys.nix~), which are then referenced by their system module.

** 🚀 Usage

*** Adding a New Host
1. Create ~hosts/new-host/configuration.nix~.
2. Define hardware and import needed profiles:
   #+begin_src nix
   imports = [
     self.nixosProfiles.base
     self.nixosProfiles.desktop
     self.users.inkpotmonkey
   ];
   #+end_src
3. Register the host in ~hosts/default.nix~ using ~mkSystem~.

*** Adding a Custom Package
1. Add the package expression to ~pkgs/[package-name]/default.nix~.
2. Register it in ~pkgs/default.nix~.
3. Access it via ~pkgs.[package-name]~.

** 🏛️ Architecture Notes

*** Why porcupineFish uses mkPiSystem?
~porcupineFish~ (and other Pi hosts) use our specialized ~mkPiSystem~. It encapsulates the ~nixos-raspberrypi~ library, which automatically handles SD card image generation and firmware/boot quirks that ~nixpkgs.lib.nixosSystem~ doesn't manage natively.

*** Why nixos-raspberrypi does not follow nixpkgs?
We pin the ~nixos-raspberrypi~ input without ~follows~ to ensure it uses the specific kernel configurations and firmware versions tested and supported by the maintainers of that hardware library.
