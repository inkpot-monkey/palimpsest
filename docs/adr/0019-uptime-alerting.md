# Uptime alerting: an off-host reachability probe plus an on-host unit-state check, into a Matrix Infra Alerts room

______________________________________________________________________

## Status: proposed

The fleet's monitoring stack is *collection-only* (VictoriaMetrics/Logs/Grafana/Vector/node-exporter store data but raise no alerts), so on 2026-06-25 a `kelpy` reboot left `stalwart`, `openclaw-gateway`, and `qbittorrent` dead for ~8h with no notification — a UID/GID reshuffle had orphaned their state (see the uid-pinning fix). We add a thin alerting tier with two complementary layers: a **reachability probe** (Gatus on `rk1b`, off-host) and a **unit-state check** (on `kelpy`), both delivering to a dedicated Matrix **#infra-alerts** room via a hookshot generic webhook. The monitored set is **monitor-by-default**, derived from the `settings.services.*` registry and enforced by a `nix flake check`, so a new service can never silently go unwatched.

## Considered Options

- **vmalert + Alertmanager on the existing VictoriaMetrics** — rejected for v1. It is the "proper" Prometheus path and the `node_systemd_unit_state` metric is already scraped, but it is the heaviest option (new rules tier + Alertmanager + an Alertmanager→Matrix payload shim), and white-box unit metrics alone reproduce the blind spots below. Revisit if the two-layer approach is outgrown.
- **A single layer (only Gatus, or only systemd state)** — rejected: each is blind exactly where the other sees, and tonight produced three different failure shapes. A reachability probe misses **port-less** services and a unit that exits *cleanly yet dead* (`stalwart`'s store-misconfig abort exits `0`, so it is `inactive`, never `failed` — an `OnFailure=`/failed-state alert would have missed the very service that triggered this). A unit-state check misses **degraded-but-running** (`qbittorrent` stayed `active` while broken). Two layers cover all three.
- **Gatus on `kelpy` (self-watching)** — rejected: a watcher on the watched host cannot report that host's death. Gatus runs on `rk1b` (the always-on voice node) for an independent vantage on `kelpy`.
- **Gatus's native Matrix provider** — rejected: reuse the hookshot generic webhook ([ADR-0017](0017-matrix-hookshot-webhooks-and-feeds.md)) so both layers share one delivery path and one room with no new Matrix account.
- **A curated allowlist / docs-only "remember to add new services" safeguard** — rejected: that *is* tonight's silent-gap failure mode, and a prose note relies on the same memory that failed. Monitor-by-default from `settings.services.*` plus a build-time guard makes "unmonitored" a build error, not a hope.
- **A self-healing / highly-available Matrix homeserver** (so alerts could keep flowing over Matrix when kelpy is down) — rejected, and explicitly out of scope. A Matrix account is **single-homed** (one authoritative homeserver), tuwunel's embedded **single-writer RocksDB** cannot run active-active, and any redundancy built *inside* the homelab still shares one **site failure domain** (power, ISP, router). The actual requirement is to be *notified*, and notification **need not be Matrix** — an off-site push channel is dramatically cheaper and is the only thing that survives a full-site outage. (Recorded so HA Matrix is not re-proposed.)

## Consequences

- **The `kelpy`-down case is handled out-of-band, not over Matrix** — because the goal is to be *notified*, and notification need not be Matrix. Matrix (`#infra-alerts` via hookshot) carries the common "host up, a service is down/degraded" alerts. When `kelpy`/`tuwunel` itself is down the alert can't cross the dead Matrix, so the `rk1b` watcher pushes to an **off-site channel** — **ntfy** (phone push, no `kelpy` in the path). **ntfy is the sole out-of-band channel.** _(Superseded by [ADR-0020](0020-out-of-band-web-push-relay.md): the out-of-band channel is realized as a self-hosted, ntfy-*compatible* web-push relay on a Cloudflare Worker, because ntfy.sh is no longer free. The rest of this ADR stands.)_ A **full-site blackout** (power/ISP, so `rk1b` can't push either) is therefore *not* covered: catching it would need an external **dead-man's switch** (a host pings out on a schedule; the external service alerts on the *silence*), the one mechanism that needs nothing at home alive — a knowingly-accepted gap for now, not built. A still-deferred nicety: an Android phone tripwire (Tasker/Automate probe → local alarm).
- **First alerting in the fleet.** This reverses the deliberate collection-only stance of the monitoring stack; the alerting tier sits *on top of* it, it does not replace it.
- **The unit-state check needs the explicit expected-up list.** It cannot simply flag "enabled but not `active`": oneshots and timer-driven jobs are `inactive` by design and would spam. The list is the `settings.services.*`-derived **expected-up** set.
- **The webhook is provisioned declaratively**, modeled on the existing hookshot provisioners (`hookshot.nix`): a oneshot writes the `uk.half-shot.matrix-hookshot.generic.hook` connection with a persisted, self-generated **deterministic hookId**, so the URL is known at eval time and both layers reference it. The provisioner can fail silently (tonight's `aionui-hookshot-provision` exited `22`), so it is itself on the monitored list.
- **Semantics are quiet by default:** alert only after **3 consecutive failed probes** (~90s, which absorbs `deploy`/`switch` restarts without a maintenance-window mechanism), send a recovery notice when a service returns, and no periodic re-alerts while still down.
- **Packaging:** the probe and unit-state check live under the existing `monitoring` profile (enabled on `rk1b` and `kelpy` respectively); the classification guard is a `nix flake check` assertion over `settings.services.*`; the webhook provisioner sits alongside `hookshot.nix`. `AGENTS.md` gains one line documenting the per-service monitor/opt-out knob.
