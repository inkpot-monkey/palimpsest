# matrix-hookshot is the front door for GitHub, generic webhooks, and feeds ŌĆö and the aionui notifier becomes a webhook shim

The fleet needed GitHub notifications in Matrix. Rather than build another bespoke poller, the matrix profile now runs [matrix-hookshot](https://matrix-org.github.io/matrix-hookshot/) (`services.matrix-hookshot`, nixpkgs 7.3.3) on kelpy as a fourth appservice alongside whatsapp and the jmap bridge. It bridges four things at once: GitHub personal notifications (the bell-icon feed, per-user OAuth), GitHub repo events (issues/PRs/CI via a GitHub App webhook), generic inbound webhooks (any service that can POST), and RSS/Atom feeds. Connections are created at runtime by talking to the `@hookshot` bot in a room ŌĆö no per-connection Nix config.

It slots into the existing declarative-appservice pattern unchanged ([ADR-0006](0006-tuwunel-matrix-homeserver.md)): a sops-rendered `hookshot-registration.yaml` is contributed to `custom.profiles.matrix.appservices`, and tuwunel loads it from `appservice_dir` like every other bridge.

## Considered Options

- **A bespoke GitHub-notifications poller** (mirroring the aionui notifier) ŌĆö rejected: it would only ever do GitHub, reusing the existing `GH_TOKEN`. Hookshot covers GitHub *plus* generic webhooks, feeds, GitLab and JIRA for later, from one service. Once we wanted "more than GitHub," the poller stopped being the cheap option.
- **Hookshot, config via the nixpkgs `settings` option** ŌĆö rejected for the secret-bearing parts: the module renders `settings` into a **world-readable Nix store** `config.yml`, but hookshot keeps the GitHub webhook secret and OAuth `client_secret` *inline* (only the App private key takes a file path). So we render the whole `config.yml` via a **sops template** and `lib.mkForce`-override `ExecStart` to consume it ŌĆö the same secret-handling stance the whatsapp/jmap registrations already take. `settings` is still set for `passFile` (the module's preStart generates the token-encryption key there).
- **Hookshot as root (module default)** ŌĆö rejected: it is internet-facing. It runs as a dedicated static system user `matrix-hookshot` with `StateDirectory` + systemd hardening. Static (not `DynamicUser`) precisely so sops can `chown` the rendered config/registration/key to it ŌĆö the classic sops-vs-DynamicUser uid clash that [the jmap bridge](0007-custom-jmap-matrix-bridge.md) sidesteps with `LoadCredential`.

## Consequences

- **Two ports, one public.** The appservice port (9993, loopback) is internal and lives in the module; the webhook/OAuth listener (9000) is registered in `parts/settings.nix` as `services.public.hookshot`, so `proxy.nix` auto-fronts `hookshot.palebluebytes.space` and the `dns` app auto-emits its record (grey-clouded, `proxy` omitted). GitHub and generic senders POST there; the OAuth callback is `ŌĆ”/oauth`.
- **The aionui notifier ([ADR-0008](0008-aionui-matrix-via-rest-poller.md)) is reduced to a webhook shim, not deleted.** aioncore still cannot push, so a poller is still required ŌĆö but in webhook mode it just POSTs each event as `{"text": ŌĆ”}` to a hookshot generic webhook (hookshot renders `text`/`html` directly, no JS transform needed). This deletes the notifier's Matrix login, room-creation, registration-token and bot-password code paths; the matrix-direct mode is retained behind `webhookUrlFile == null` for portability. Net: one fewer Matrix bot, one fewer self-registered account, one fewer secret.
- **A runtime bootstrap step the deploy can't encode.** The generic webhook URL only exists after the webhook is created via a bot command, so `aionui_hookshot_webhook_url` starts empty in sops and the notifier *idles* until it's populated (then picks it up without a restart). The GitHub App is likewise un-Nix-able: GitHub has no API (nor `gh` command) to create an App, so it is created once by hand in the GitHub UI (its id/private-key/webhook-secret/OAuth-client land in `secrets/profiles/matrix.yaml` as `hookshot_github_*`, alongside the random `hookshot_as_token`/`hookshot_hs_token`). The App's webhook URL is `ŌĆ”/github/webhook` (the bare `/` path is deprecated) and its OAuth callback is `ŌĆ”/oauth`. For personal notifications alone the App needs no installation ŌĆö the OAuth `github login` user token already spans every account/org you can see; installing it (personal or org) is only for repo-event firehose rooms.
- **passkey.pem must persist.** It encrypts hookshot's stored GitHub/OAuth tokens; losing it forces re-auth of every connection. The whole `/var/lib/matrix-hookshot` state dir is added to impermanence.
- **Config style is bot-commands** (`github login`, `github notifications toggle`, `webhook`, `feed`), so neither the provisioning widget nor metrics listeners are exposed ŌĆö the public listener serves only the `webhooks` resource.
