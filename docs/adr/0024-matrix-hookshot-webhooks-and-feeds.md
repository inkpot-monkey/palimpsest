# matrix-hookshot is the front door for GitHub, generic webhooks, and feeds — and the aionui notifier becomes a webhook shim

> **Update (2026-06-25):** AionUi and its notifier were removed (see [ADR-0025](0025-claude-relay-matrix-interface.md)), so the `aionui-hookshot-provision` oneshot + `#aionui-alerts` connection described below are gone. The hookshot core — GitHub notifications/repo-events, generic webhooks, feeds — is unchanged and remains.

The fleet needed GitHub notifications in Matrix. Rather than build another bespoke poller, the matrix profile now runs [matrix-hookshot](https://matrix-org.github.io/matrix-hookshot/) (`services.matrix-hookshot`, nixpkgs 7.3.3) on kelpy as a fourth appservice alongside whatsapp and the jmap bridge. It bridges four things at once: GitHub personal notifications (the bell-icon feed, per-user OAuth), GitHub repo events (issues/PRs/CI via a GitHub App webhook), generic inbound webhooks (any service that can POST), and RSS/Atom feeds. Connections are created at runtime by talking to the `@hookshot` bot in a room — no per-connection Nix config.

It slots into the existing declarative-appservice pattern unchanged ([ADR-0006](0006-tuwunel-matrix-homeserver.md)): a sops-rendered `hookshot-registration.yaml` is contributed to `custom.profiles.matrix.appservices`, and tuwunel loads it from `appservice_dir` like every other bridge.

## Considered Options

- **A bespoke GitHub-notifications poller** (mirroring the aionui notifier) — rejected: it would only ever do GitHub, reusing the existing `GH_TOKEN`. Hookshot covers GitHub *plus* generic webhooks, feeds, GitLab and JIRA for later, from one service. Once we wanted "more than GitHub," the poller stopped being the cheap option.
- **Hookshot, config via the nixpkgs `settings` option** — rejected for the secret-bearing parts: the module renders `settings` into a **world-readable Nix store** `config.yml`, but hookshot keeps the GitHub webhook secret and OAuth `client_secret` *inline* (only the App private key takes a file path). So we render the whole `config.yml` via a **sops template** and `lib.mkForce`-override `ExecStart` to consume it — the same secret-handling stance the whatsapp/jmap registrations already take. `settings` is still set for `passFile` (the module's preStart generates the token-encryption key there).
- **Hookshot as root (module default)** — rejected: it is internet-facing. It runs as a dedicated static system user `matrix-hookshot` with `StateDirectory` + systemd hardening. Static (not `DynamicUser`) precisely so sops can `chown` the rendered config/registration/key to it — the classic sops-vs-DynamicUser uid clash that [the jmap bridge](0007-custom-jmap-matrix-bridge.md) sidesteps with `LoadCredential`.

## Consequences

- **Two ports, one public.** The appservice port (9993, loopback) is internal and lives in the module; the webhook/OAuth listener (9000) is registered in `parts/settings.nix` as `services.public.hookshot`, so `proxy.nix` auto-fronts `hookshot.palebluebytes.space` and the `dns` app auto-emits its record (grey-clouded, `proxy` omitted). GitHub and generic senders POST there; the OAuth callback is `…/oauth`.
- **The aionui notifier ([ADR-0008](0008-aionui-matrix-via-rest-poller.md)) becomes webhook-only and fully declarative.** aioncore still cannot push, so a poller is still required — but it now only POSTs each event as `{"text": …}` to a hookshot generic webhook (hookshot renders `text`/`html` directly, no JS transform needed). The notifier's entire Matrix side — login, registration token, bot password, room creation — is **deleted**, not kept as a fallback. Net: one fewer Matrix bot, one fewer self-registered account, no notifier secret.
- **The aionui connection is provisioned declaratively, no bot command.** A `aionui-hookshot-provision` oneshot reuses the hookshot appservice `as_token` to create the `#aionui-alerts` room and the generic-webhook connection. Hookshot stores a generic hook's id in the bot's *room account data* (mapping `hookId → stateKey`) and the connection in a `uk.half-shot.matrix-hookshot.generic.hook` state event; the provisioner writes both with a **persisted, self-generated secret hookId**, so the webhook URL (`…/webhook/<hookId>`) is deterministic and is written to the file the notifier reads. (Hookshot also supports config-defined "static" connections, where the state key *is* the hookId — but those need the `roomId` at build time, which a server-generated id can't satisfy; provisioning at runtime via the API sidesteps that.) The dependency on hookshot's internal account-data/state schema is the tradeoff for zero-touch.
- **The GitHub App is the one un-Nix-able bootstrap.** GitHub has no API (nor `gh` command) to create an App, so it is made once by hand in the UI; its id/private-key/webhook-secret/OAuth-client land in `secrets/profiles/matrix.yaml` as `hookshot_github_*`, alongside the random `hookshot_as_token`/`hookshot_hs_token`. The App's webhook URL is `…/github/webhook` (the bare `/` is deprecated) and its OAuth callback is `…/oauth`. For personal notifications alone the App needs **no installation** — the OAuth `github login` user token already spans every account/org you can see; installing it (personal or org) is only for repo-event firehose rooms.
- **passkey.pem must persist.** It encrypts hookshot's stored GitHub/OAuth tokens; losing it forces re-auth of every connection. The whole `/var/lib/matrix-hookshot` state dir is added to impermanence.
- **GitHub/feeds connections are still bot-command driven** (`github login`, `github notifications toggle`, `feed`); GitHub *notifications* can't be declarative anyway (per-user OAuth). Neither the provisioning widget nor metrics listeners are exposed — the public listener serves only the `webhooks` resource.
